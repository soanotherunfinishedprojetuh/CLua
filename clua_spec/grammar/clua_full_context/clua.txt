1 Introduction

1.1 Motivation

LuaU is widely used for Roblox game development due to its simplicity and tight integration with the Roblox engine. 
However, large-scale projects often encounter limitations related to static analysis, data layout control, and 
serialization boilerplate.

TypeScript-to-LuaU transpilers improve maintainability by introducing static typing, but they are 
primarily designed around the JavaScript ecosystem rather than the specific requirements of 
LuaU-based game development. As a result, they do not provide direct control over low-level 
code generation, data layout, or LuaU-specific optimization patterns.

CLua is designed specifically for the LuaU runtime. 
Its goal is to provide a statically typed language that compiles to highly optimized LuaU code 
while enabling explicit control over serialization, memory layout, and compile-time code generation.

The language prioritizes three primary objectives:
-Performance: generated LuaU code should follow patterns that are efficient for the LuaU VM.
-Maintainability: large projects should benefit from strong typing and predictable data layouts.
-Developer tooling*: the compiler should provide diagnostics and utilities that compensate for 
limitations in the Roblox debugging ecosystem.

1.2 What is CLua?
CLua is a statically typed language, with meta programming layer on top. This language ensures that 
data like structures are as compact as possible in memory while balancing for performance.
User of course will have also control over layout with special attributes provided or defined in 
the metaprogramming language. 

The metaprogramming language is just CLua itself extended by attributes but is also executed at compile time, 
any values that will be generated at compile time, can be used at runtime if user states that explicitly (something as simple as: "let clua_runtime = @clua_comptime_var;" with caveats though).   

2. Design constraints and requirements

For CLua to be a consistent language and be easily mapable to LuaU,
while ensuring almost full or full compile time analysis, there 
must be a set of constraints and requirements to what CLua is going to 
look like. Each constraint and requirement must justify why such decision
was made. 

2.1 Requirements
R1: CLua must be statically typed.

Justification:
Static typing allows the compiler to determine field sizes and perform
compile-time validation.
---
R2: CLua must support function callbacks.

Justification:
Callbacks are required for event-driven programming and task systems.
Callbacks must be transferable between threads.
---
R3: CLua must support LuaU's parallel execution model.

Justification:
Roblox provides a parallel execution environment based on isolated
LuaU threads. CLua must generate code compatible with this model.
---
R4: CLua must allow embedding of native LuaU code.

Justification:
Some LuaU features cannot be expressed directly in CLua.
Embedding allows developers to access the full LuaU runtime when needed.
---
R5: All POD types must be serializable.

Justification:
Serialization is required for network communication and thread messaging.
---
R6: CLua must allow explicit control of data layout.

Justification:
Predictable layouts are required for efficient serialization and
buffer-based memory storage.
---
R7: Generated code must be valid LuaU.

Justification:
CLua is a source-to-source compiler targeting the LuaU runtime.
---
R8: CLua provides native primitives from LuaU

Justification:
This makes for compiler more explicit that a certain data type is being used 
and allows for better optimizations.
---
R9: CLua must clearly distinguish between POD and non-POD types.

Justification:
Serialization, buffer mapping, and thread communication require
deterministic memory layouts. Only POD types can safely participate
in these operations.

2.2 Constraints
C1: LuaU represents numeric values using double-precision floating point numbers.

Implication:
CLua integer types smaller than 64 bits may require wrapping or masking
to preserve deterministic behavior.
---
C2: LuaU imposes a limit on the number of local variables within a single function.

Implication:
The CLua compiler must ensure that generated LuaU code does not exceed
this limit, potentially by splitting large functions.
---
C3: LuaU tables serve both as arrays and hash maps.

Implication:
CLua must provide explicit control over data layout to avoid unintended
performance penalties caused by hash-table usage.
---
C4: LuaU functions and closures cannot be serialized between threads.

Implication:
CLua function pointers must be represented as callback identifiers
rather than raw closures.
---
C5: LuaU buffers provide a contiguous block of memory but require manual
interpretation of stored data.

Implication:
CLua must provide mechanisms to map structured data onto buffers.

4.4 Array Types
    All arrays in CLua are explicitly dynamic, but there are
    two ways to create an array in CLua and both have 
    different behaviours.

    LuaU Array: 
        let int_array: ArrayPointer = Array<I32>(50);
        int_array.push_top(49);
        print(int_array[50]); --Client: 49
    CLua Style Array:
        let memory_region = new<I32>(50);
        memory_region[49] = 50;

        let extended_array = new<I32>(51);
        //you need to copy data from previous array to extend it, or just allocate enough memory
        memory_region[49] = 50;
        memory_region[50] = 51;

        print(extended_array[50]); --Client: 51
        delete memory_region;
        delete extended_array;
